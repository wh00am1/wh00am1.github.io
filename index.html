hackme.inndy.tw onepunch Writeup

發表於 2020-10-16

hackme.inndy.tw onepunch (pwn)

challenge

nc hackme.inndy.tw 7718

Punch!

保護機制

Partial RELRO, NX, Canary

分析

程式邏輯

main:

先是讀一個pointer和int, 接著把pointer指向的地方改成輸入的數字

因此我們拿到了一個Arbitrary Write, 但只有一次

先補一下assembly code:

補充知識

然後再補一下assembly jxx <address> instruction:

基本上是2 byte的instruction:

  1. 第一個byte是jxx指令本身
  2. 第二個byte是offset(相對於當前的RIP/EIP)

以0x00400767 的jne 0x400773為例:

  1. 此處機械碼為 750a
  2. 0x75 為 jne
  3. 0x400773 = 0x400767 + 0xa

所以我們可以先把0x00400767 的jne 0x400773 改成跳到main

這樣就可以成為無限迴圈(因為v4永遠不會等於255, 0xff 是無效的opcode)

利用方式

有了無限迴圈之後, 就可以寫值了

只要最後把它再次patch回去就好

因為可以將v6的值改成 v4, 我們可以逐byte寫入shellcode(注意v4是int)

接下來是要想寫到哪一段記憶體, 因為有NX, 可以考慮寫到.text

以我的exploit為例, 是跳轉到__libc_csu_init 因為空間夠大而且address > main 0x400773

exploit

注意最後要再patch回來

p.s. 寫到csu是我看別人writeup之後改的

pwnable.tw ORW Writeup

發表於 2020-10-16

pwnable.tw ORW (pwn)

保護機制

Canary, Partial RELRO

分析

main

先是讀取了一大段shellcode, 接著直接跑起來

不過還是先看看 orw_seccomp():

orw_seccomp

在C語言有一種”SandBox模式”, 限制程式只能使用 open(), read(), write()

可以通過seccomp() 或是prctl()開啟

所以這裡的邏輯不重要, 只要知道只能 open, read, write就好

也不用繞canary, 因為沒有Control PC

exploit

利用pwntools shellcraft()拼出call open, read, write印出flag

pwnable.tw start Writeup

發表於 2020-10-16

pwnable.tw start (pwn)

保護機制

Nope

分析

程式僅有一OEP (entry 0)

簡單看一下:

先是把字串"Let's start the CTF:"印出, 接著用read()讀了 60 bytes造成overflow

然後就沒了, 非常簡單

接著就是想辦法用 ret2shellcode拿到shell, 但stack位置無從得知, 必須想辦法找Gadget leak stack

仔細觀察組語可以發現在0x08048087處一路執行下去, 可以用write把ESP印出來

註: 人肉decompile大概可以看出是

write(1, $esp, 0x14)

exploit

先控制EIP跳轉到0x08048087後, 再跳轉回主程式, 將shellcode寫到stack上然後getshell

小坑點:

  1. 因為輸入長度只有60, 不能使用shellcraft, 可以到shellstorm上面找x86 shellcode
  2. 在堆leak chain的時候為了穩定, 使用send(), 而不是sendline()

hackme.inndy.tw Rsbo Writeup

發表於 2020-10-15

hackme.inndy.tw Rsbo(&Rsbo 2) (pwn)

challenge

Tips: ROP, open, read

保護機制

NX開啟，Partial RELRO

分析

main的邏輯:

可以看到是先分配了長度60的buf, 以及4個整數，接著呼叫alarm(), init()

先看看init()的邏輯:

init()的邏輯主要是:

  1. 打開/home/ctf/flag, 然後讀到buf (長度16)
  2. 搞出一個seed, 然後對flag做了一些操作

看不出有明顯漏洞, 接著看看read_80_bytes():

很簡單, 就是讀0x80 bytes

接著題目說 Open, Read, 就看不太懂了

不過觀察組語可發現 main() stack大小小於0x80(128), 而read_80_bytes() 造成 overflow

因為ASLR, NX 的關係, 需要想辦法ROP

但binary太小, 組不起ROP chain, 因此使用libc

只要Leak GOT, 接著就可以算出libc base & getshell了
